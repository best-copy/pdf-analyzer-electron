# 작업 내역 — 2026-07-01

편집(레이아웃·회전)과 적용이 **끊김없이 부드럽게, 빠르게** 진행되도록 실시간
파이프라인을 최적화. 버전 **1.0.0 → 2.0.0**.

---

## 0. 배경 — 실시간 편집 핫패스

편집 사이드바에서 슬라이더를 움직이거나 머리글/바닥글을 입력할 때마다 실행되는 흐름:

1. `scheduleLivePreview()` (120ms 디바운스) → `runLivePreview()`
2. `buildBaseProcessed()` — `baseSignature()`(순서·회전·흑백)로 캐시. 레이아웃만 바꾸면 히트.
3. `applyLayoutTransform()` → `worker-assemble.js`(N-up·규격·테두리·머리글바닥글·워터마크)
4. `renderProcessedPreview()` → pdf.js로 결과 렌더

병목 두 곳을 찾음: (3) 워커가 매 편집마다 base PDF 전체를 재파싱, (4) pdf.js 문서 미해제.

## 1. 워커 base 파싱 캐시 (레이아웃 편집)

- **문제:** `handleLayoutTransform`가 편집 한 번마다 `PDFDocument.load(srcBytes)`로 base
  PDF 전체를 다시 파싱 + 모든 페이지 재-embed. 레이아웃 옵션만 바꿔도 base는 동일한데 낭비.
- **해결:** `worker-assemble.js`가 `payload.baseSig`(=메인 `baseSignature()`)를 키로 파싱한
  `src` 문서를 `self.__srcCache = {sig, doc}`에 **1개** 보관. base가 그대로면 재파싱 생략.
  - 라이브 미리보기는 직렬(`_liveRunning`)이고 워커풀이 방금 반납한 워커를 LIFO로 다시
    꺼내므로 **같은 워커가 연속 처리 → 캐시 거의 항상 적중**(라우팅 변경 불필요).
  - base가 바뀌면(회전 등) sig가 달라져 새로 파싱. src는 embedPage 대상 읽기 전용이라
    여러 출력 문서가 공유해도 안전(node로 동일 src를 독립 out 4개에 embed → byte-identical 확인).
  - 메인 `applyLayoutTransform`이 `baseSig`를 payload에 추가. 다운로드 경로
    (`buildOptimizedOutput`)는 baseSig 없이 호출 → 캐시 미적용(1회성이라 무방).

## 2. pdf.js 문서 해제 (미리보기 렌더)

- **문제:** `renderProcessedPreview`가 매 갱신 `getDocument`로 결과 PDF를 새로 파싱하는데
  `destroy()`를 안 해 pdf.js 워커 메모리가 세션 내내 누적 → 편집을 반복할수록 전체가 느려짐.
- **해결:** `let pdf = null` 호이스트 + `finally { pdf.destroy() }`로 해제
  (토큰 불일치 조기 return에도 정리됨).

## 3. 회전 편집 최적화 — 조립본 재사용

- **문제:** 페이지 회전 시 `r.rotation`이 바뀌어 `baseSignature()` 무효화 →
  `buildBaseProcessed()`가 메인 스레드에서 **전체 재조립**(`load` + `copyPages` + 흑백조립
  + `save`). `/Rotate` 플래그 하나 바꾸려고 문서 전체를 다시 만들어 큰 문서에서 순간 멈춤.
- **해결 — 회전 전용 빠른 경로:**
  - 조립된 `outDoc`을 **회전을 뺀 키**(`baseOrderBwKey()` = 순서 + 흑백)로 `_baseAssembled`에
    캐시하고, 페이지별 원본 각도 `baseAngle`(원본 /Rotate)을 기록.
  - 순서·흑백이 그대로고 **회전만 바뀌면** → 캐시된 `outDoc` 재사용, 각 페이지
    `setRotation((baseAngle + 요청각) % 360)`만 다시 걸고 재저장. **`load`·`copyPages`·
    흑백조립 전부 생략.**
  - `baseAngle` 기준이라 회전이 누적되지 않고 항상 `(원본각 + 요청각)`으로 idempotent하게 반영.
  - 재정렬·흑백선택·빈 페이지 삽입은 키가 바뀌어 정상적으로 전체 재조립.
  - `clearProcessCaches`(탭 전환)에서 `_baseAssembled` 해제.

## 4. 검증

- **Node 헤드리스**(vendored pdf-lib 사용):
  - 동일 `src`를 4개 독립 출력에 `embedPage` → 매번 byte-identical(원본 무변형 확인).
  - 같은 `outDoc`에 반복 `save()` + 재-`setRotation` → `[0,90]→[90,180]→[180,270]→[0,90]`
    정확(baseAngle 기준 idempotent, 리셋 시 원본 회전 복원).
- `node --check src/worker-assemble.js` 통과, `index.html` 인라인 스크립트 파싱 통과.

## 5. 남은 병목 (미착수)

- 재저장(`outDoc.save`) 자체와 재정렬/흑백선택 시 전체 재조립은 여전히 메인 스레드.
  `outDoc`은 워커로 transfer 불가라 `save` 오프로드는 구조적으로 어려움.
- 필요 시 렌더 가상화(뷰포트 근처 페이지만 우선 렌더) 등 별도 접근 검토 가능.

## 6. 빌드/배포

- 메모리 지침대로 **포터블(win-unpacked)만 갱신**: `src/index.html`,
  `src/worker-assemble.js`, `package.json`을 `dist/win-unpacked/resources/app/`에 동기화.
- zip 재생성·설치형 빌드는 안 함(요청 시에만).
