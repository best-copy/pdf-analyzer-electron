# 작업 내역 — 2026-06-23

한컴 한글 HWP → PDF 변환 시 **용지 크기가 문서 크기대로 안 나오는 문제** 조사 및 조치.
대상 파일: `D:\바탕화면\윤임수 논문(202606최종).hwp` (37MB, 138p, 신국판 190×260).

---

## 1. 증상

- 포터블 앱으로 변환하면 문서(190×260)가 **A4로 생성**됨.

## 2. 원인 분석 (단계적)

1. **직접 원인 = 이전 세션에 추가한 자동 보정 코드**
   - `main.js` 의 `correctInflatedPageSize()` 가 변환 결과를 "표준 용지로 스냅" 하면서
     이 문서를 A4(210×297)로 만들고 있었음. (당시 한글이 210×500으로 부풀려 내보낸 걸
     A4로 스냅) → **잘못된 접근이라 전량 제거.**

2. **정상 문서는 크기 정확** — 새로 만든 190×260 → 190×260, 250×180 → 250×180,
   A4/A3/가로 모두 정확히 변환됨(문서 주도).

3. **이 논문 파일만 틀어짐** — HWPX로 풀어 확인하니 4개 구역 전부
   `<hp:pagePr landscape="WIDELY" width=190mm height=260mm>` —
   **세로 치수인데 가로(WIDELY) 방향 플래그가 박힌 모순 상태.**
   이 이상 때문에 한글 PDF 내보내기가 문서 크기를 못 잡고
   **시스템/프린터 전역 용지로 폴백**(A4 또는 210×500)함.
   콘텐츠는 용지와 무관하게 **고정 189×270mm 스케일**로 얹힘.

4. **자동화로 강제 불가 확인**
   - COM `Execute("PageSetup")` 으로 크기 강제 → 오히려 A4로 리셋 + 대화상자 hang.
   - 네이티브 SaveAs / HWPX landscape 직접 편집(NARROWLY) → 해결 못 함(후자는 악화).
   - 부풀린 출력은 콘텐츠가 스케일돼 사후 크롭으로 1:1 복원 불가.
   - (주의) PageSetup 으로 만든 테스트 문서가 **전역 용지를 오염**시켜 anomalous 문서가
     그 크기를 상속함 — 진단 중 자가 오염 주의.

## 3. 조치

- `main.js`: `correctInflatedPageSize` 및 관련 헬퍼(`measureInkBoxes`/`pickStandardPaper`/
  `isStandardSize`/STD_PAPERS/pdf-lib require) **전부 제거**, 두 변환 콜백 원복.
- `src/pdf_inkbbox.ps1` (잉크 영역 측정용으로 추가했던 스크립트) **삭제**.
- `src/convert_hwp.ps1` 는 **원본 그대로 유지**(정상 문서는 이미 올바르게 변환됨).
- 포터블 앱(`dist/win-unpacked/resources/app/`)에 `main.js` 동기화, `pdf_inkbbox.ps1` 제거.
- 검증: 깨끗한 190×260 문서가 현재도 190×260으로 정상 변환됨.

## 4. 권장 (anomalous 문서)

방향 플래그가 깨진 문서는 한글에서 **F7(편집 용지) → 세로 + 190×260 → 문서 전체 적용 →
저장** 후 변환하면 정상. 문서 자체를 한 번 바로잡는 게 가장 깨끗함.

## 5. 한글 GUI/인쇄 기반 PDF 저장 방식 (토글 옵션 추가)

사용자 요청으로 **"Microsoft Print to PDF" 프린터로 인쇄**하는 대체 변환 경로를 구현.

- 신규 파일 `src/convert_hwp_gui.ps1`: 한글 COM `Print` 액션으로 MS Print to PDF 에
  **파일 직접 인쇄**(`PrintToFile=1` + `FileName`). **대화상자 없이** PDF 생성 성공.
- `main.js` 에 전환 스위치 추가:
  ```js
  const HWP_CONVERT_MODE = 'api';  // 'api'=convert_hwp.ps1(기본), 'print'=convert_hwp_gui.ps1
  ```
  롤백: 값을 `'api'` 로 두면 기존 방식. 시도: `'print'` 로 변경 후 앱 재시작.

### 테스트 결과 / 결론
- **인쇄 경로는 출력 용지가 항상 프린터 기본 용지(A4)로 고정**됨(검증: 190×260 문서 → 210×297).
  HPrint 파라미터셋에 용지 크기 필드가 없어(`PaperWidth/Height/Kind` 모두 빈 값) 문서 크기 지정 불가.
- 즉 **GUI/인쇄 방식은 문서 크기를 유지하지 못함**(정상 문서엔 오히려 API 방식보다 나쁨).
- → **기본값은 `'api'`(현재 방식) 유지.** GUI/인쇄 방식은 실험용으로만 토글 가능하게 남겨둠.

### 빌드/배포
- 메모리 지침대로 포터블(win-unpacked)만 갱신: `main.js`, `src/convert_hwp_gui.ps1` 동기화.
  (`pdf_inkbbox.ps1` 은 포터블에서도 제거)

### 6. 롤백 (사용자 요청)
- GUI/인쇄 방식이 문서 크기를 유지 못 함을 확인 → **실험 전량 롤백**.
  - `main.js`: `HWP_CONVERT_MODE`/`HWP_SCRIPTS` 토글 제거, `convert_hwp.ps1` 단독 사용으로 원복.
  - `src/convert_hwp_gui.ps1` 삭제(포터블 포함).
- **최종 상태: 현재(API) 변환 방식만 사용** — 정상 문서는 문서 크기 그대로 변환됨.

## 7. 앱 아이콘 교체 + 색상 조정

사용자 요청으로 앱 아이콘 교체.

- 원본: `D:\바탕화면\pdf-file.ico` (PDF 문서 모양, **검정 단색 실루엣**).
- `main.js` 가 `src/icon.ico` 를 참조하나 파일이 없었고, package.json 에도 win 아이콘 설정이
  없었음 → 새 아이콘을 `src/icon.ico` 로 복사 + `package.json` `build.win.icon: "src/icon.ico"` 추가.
- **검정색은 어두운 작업표시줄/배경에서 거의 안 보이는 문제** → PDF 대표색 **빨강(#D32F2F)** 으로
  재색칠(알파 유지). 16/32/48/64/128/256px 6개 해상도를 담은 정식 `.ico` 로 재생성.
- 포터블 앱 재빌드(`npm run build`, dir 타깃)로 `dist/win-unpacked/PDF 분석기.exe` 에 새 아이콘 적용.
  - 빌드 시 `CSC_IDENTITY_AUTO_DISCOVERY=false` 로 코드사인 시도 비활성화.
- 메모리 지침대로 포터블(win-unpacked)만 갱신, zip 재생성·설치형 빌드는 안 함.
