### 3-1. 파일 선택 버튼 / 드래그 미작동 (3개 원인)

### 3-2. PDF.js 워커 asar 로드 실패
- **원인**: 패키지 내 `app.asar` 안의 파일은 Web Worker로 직접 로드 불가
- **수정**: preload의 `fs.readFileSync`로 워커 파일 내용 읽기 → **Blob URL** 생성 → `workerSrc`에 적용

### 3-3. 대용량 PDF IPC 전송 실패
- **원인**: 파일 전체를 JSON 배열로 IPC 전달 → 50MB+ PDF에서 느리고 불안정
- **수정**: main은 **파일 경로만 반환**, preload의 `readFile(path)`로 `ArrayBuffer` 직접 반환

### 3-4. 내보내기 후 업무 관리 앱에 견적서 미표시

- 추가: 이미 열려 있으면 새 창 대신 기존 창 포커스

## 5. 2026-06-04 추가 작업

### 5-1. Dot Gain 선택 드롭다운 제거
- 처리 옵션에서 Dot Gain 선택 UI(`#dotGainSelect`) 삭제

### 5-2. 버그 수정: 변환 후 PDF 열리지 않는 오류

**원인:** pako가 단일 청크 출력 시 내부 버퍼의 subarray를 반환
- `deflated.length` (실제 압축 크기) < `deflated.buffer.byteLength` (내부 버퍼, 최대 16KB)
- worker에서 `deflated.buffer` 전체를 전송 → 메인 스레드에서 잘못된 `Length` 기록
- PDF 리더가 스트림을 올바르게 파싱하지 못해 파일이 열리지 않음

**수정:** `worker-gray.js` — `jpeg2gray`, `flate2gray`, `icc-flate2gray` 3곳
```js
// 수정 전
self.postMessage({ deflated: deflated.buffer, ... })

// 수정 후
const deflatedBuf = deflated.buffer.slice(deflated.byteOffset, deflated.byteOffset + deflated.length);
self.postMessage({ deflated: deflatedBuf, ... })
```

### 5-3. 버그 수정: Acrobat 페이지 표시 오류

**원인:** FlateDecode 이미지 변환 후 `Filter`/`DecodeParms` 형태 불일치
| Filter | DecodeParms | 유효 여부 |
|--------|-------------|----------|
| `/FlateDecode` (이름) | `<< ... >>` (딕셔너리) | ✅ |
| `[/FlateDecode]` (배열) | `[<< ... >>]` (배열) | ✅ |
| `[/FlateDecode]` (배열) | `<< ... >>` (딕셔너리) | ❌ 기존 코드 |

일부 PDF에서 `Filter`가 배열 형태로 저장되어 있는데, 변환 후 `DecodeParms`만 딕셔너리로 바꾸고 `Filter`는 배열로 남아 Acrobat이 이미지 디코딩 실패.

**수정:** `index.html` — `convertFlateXObjectToGrayscale` (일반 + ICCBased 경로)
- `Filter`를 `/FlateDecode` (이름)으로 명시 정규화
- `BitsPerComponent: 8`, `Width`, `Height` 명시 갱신 추가

### 5-4. 버그 수정: Acrobat 페이지 표시 오류 (2차) — 명명된 색공간 처리

**원인 1 — 명명된 색공간 이미지 미변환**

InDesign 등 전문 DTP 도구로 만든 PDF는 이미지의 ColorSpace를
`/DeviceRGB` 직접 명시 대신 `/Cs1` 같은 명명된 색공간으로 지정.
`convertFlateXObjectToGrayscale`이 이 형태를 인식 못해 변환 없이 리턴.

**원인 2 — 미변환 이미지의 색공간이 Resources에서 삭제됨**

③ 색공간 정리 단계가 변환 실패한 이미지가 아직 참조 중인 색공간도
Resources.ColorSpace에서 삭제 → Acrobat이 해당 이미지 렌더링 불가
→ "페이지를 표시할 수 없음" 오류

**수정 내용 (`index.html`)**

1. `resDict`를 `convertXObjectImageToGrayscale` → `convertFlateXObjectToGrayscale`로 전달
2. 명명된 색공간 해결: csObj가 `/Cs1` 형태면 `resDict.ColorSpace`에서 실제 정의(`[/ICCBased ...]` 등)로 교체 후 기존 변환 경로 재사용
3. ③ 색공간 정리: XObject가 아직 참조 중인 색공간은 삭제 보류
   ```
   stillUsedCSNames = {XObject들의 ColorSpace 이름 수집}
   toDelete에서 stillUsedCSNames에 있는 항목 제외
   ```

### 5-5. 버그 수정: Acrobat 페이지 표시 오류 (3차) — Pattern 색공간 처리

**원인 분석 (첨부 PDF 직접 분석)**

`2025학년도 2학기_간호대학 사본.pdf`를 Node.js + pdf-lib로 분석한 결과:
- 이미지 XObject 없음 (순수 벡터 PDF)
- 6,295개 Form XObject (투명도 그룹 3,263개)
- Resources.ColorSpace:
  - `/CS0` → `[/ICCBased ...]` — ICC 색공간
  - `/CS1` → `[/Separation ...]` — 별색
  - **`/CS2` → `[/Pattern]`** — 패턴 색공간 (그래디언트 34개)
- 컨텐츠 스트림: `/CS2 CS /P0 SCN` (패턴 채색)

**버그 재현 경로:**
1. 코드가 `/CS2 CS` 제거 → strokeCS만 기록하고 연산자 삭제
2. `/P0 SCN` (패턴 이름 인수)은 숫자 정규식과 불일치 → **변환 없이 스트림에 잔존**
3. ③ 색공간 정리가 `/CS2`(Pattern)를 Resources에서 삭제
4. Acrobat: Pattern CS 없는 상태에서 `/P0 SCN` 실행 → **페이지 렌더링 오류**

**수정 내용 (`index.html`, `worker-gray.js`)**

1. csGrayMap 빌드 시 Pattern 타입 색공간 기록: `csGrayMap[name] = { type: 'Pattern' }`
2. `grayifyStream`: Pattern 타입 CS 연산자 제거 않고 유지
   ```
   /CS2 CS → /CS2 CS 그대로 (Pattern이므로)
   /P0 SCN → 변환 없이 유지 (숫자가 아닌 이름 인수)
   ```
3. ③ 색공간 정리: `/Pattern` 색공간은 삭제 대상에서 제외
   - Pattern 색공간 보존 → Acrobat이 정상 렌더링

### 5-6. 버그 수정: Pattern 색공간 오류 (4차) — csGrayMap 키 불일치

**원인**

5-5 수정에서 grayifyStream의 Pattern 판별 코드에 키 불일치 버그 존재:
```javascript
// 버그: csGrayMap['CS2']  → undefined (키는 '/CS2'로 저장됨)
const info = csGrayMap && csGrayMap[name];

// 수정: csGrayMap['/CS2'] → { type: 'Pattern' } (정상)
const info = csGrayMap && csGrayMap['/' + name];
```

| 케이스 | 수정 전 | 수정 후 |
|--------|---------|---------|
| `/CS1 CS` (Separation) | REMOVE ✓ | REMOVE ✓ |
| `/CS2 CS` (Pattern) | REMOVE ✗ | KEEP ✓ |

CS2(Pattern)가 제거되면 `/P0 SCN` 등 패턴 호출 앞에 활성 Pattern 색공간이
없어지므로 Acrobat이 렌더링 실패.

**수정 (`worker-gray.js`):** `csGrayMap[name]` → `csGrayMap['/' + name]`

   - SMask Form 내부 리소스 재귀 처리 (`processXObjects`, `processExtGState`)

## 5-8. 버그 수정: 콘텐츠 스트림 내 인라인 이미지 칼라 잔류 (2026-06-07)

**증상:** 흑백변환 후에도 특정 구간(선이 겹치는 부분)에 파란색/보라색 작은 이미지가 남음

**원인 분석 (original.pdf 구조 파악):**
- PDF 페이지 → Form XObject (obj 8) → 677개 RGB 래스터 이미지 XObject (별도 obj)
- Form XObject의 콘텐츠 스트림(1.3MB 비압축)에 **541개의 인라인 이미지(BI/EI)** 포함
- 인라인 이미지 예시: `/W 41 /H 9 /BPC 8 /CS /RGB` + 바이너리 RGB 픽셀 데이터
- 픽셀값 `8D 84 B7` = (141, 132, 183) = 파란-보라색 → 스크린샷의 파란 선 원인

**기존 코드 문제:**
- `grayifyStream`은 `(...)`, `<...>` 문자열 리터럴만 보호하고 `BI`/`EI` 블록은 완전 무시
- 바이너리 픽셀 데이터가 regex에 의해 color operator로 오인식될 위험도 있음
- 인라인 이미지는 XObject 처리 코드와 완전히 별개 → 변환 대상에서 누락

**수정 (`worker-gray.js`):**
- `grayifyStream`의 문자 스캐너 while 루프에 BI 감지 분기 추가
- `BI`가 줄 첫 머리에 나타나면:
  1. 딕셔너리(`biGetNum`, `biGetCS`, `biGetF`) 파싱 → W, H, BPC, CS, F 추출
  2. **비압축 RGB/CMYK (Filter 없음)**: W×H×channels 바이트 읽어 BT.601 그레이 변환, `/CS /G`로 교체
  3. **이미 그레이 또는 알 수 없는 CS**: 바이너리 데이터를 regex로부터 보호하며 그대로 통과
  4. **필터 있음(압축)**: heuristic으로 EI 탐색 후 그대로 통과 (미구현, 안전 처리)
